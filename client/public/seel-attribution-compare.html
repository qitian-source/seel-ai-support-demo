Seel — 评价归因方案对比

免费版（抓取 + 模糊匹配）vs Trustpilot Business API（referenceId 精确归因）

😊 用户在 CSAT 中选择 Satisfied
两种方案在此分叉 — 后续处理逻辑完全不同
🆓
免费版 / Free Tier
依赖公开页面爬取 + 多级模糊匹配算法
1
写入意图日志 (Intent Log)
Seel 插件本地记录用户信息，等待后续匹配
orderId: "10267"
expectedUsername: "Jennifer Martinez"
csatResponse: "satisfied"
2
每日定时爬取 TP 公开主页
抓取商户 Trustpilot 页面最近 72h 评价
⚠️ 无 API 权限，只能解析公开 HTML
🔍 三重匹配算法（优先级递减）
1
Exact Name Match
authorName === expectedUsername（完全一致）
100
2
Fuzzy Name Match
"D. Wang" → "David Wang" · Token + 缩写推断
55–99
3
Keyword Match
评价文本包含 "#10267" 等订单号
90
⚠️ 模糊归因结果
置信度 55–100%，存在误判 匹配成功 + rating ≥ 4 → $5
💎
Trustpilot Business（付费版）
官方 Invitation API · referenceId 精确绑定
1
调用 Trustpilot Invitation API
Seel 直接向 TP 发送邀请请求，orderId 作为 referenceId 绑定
POST /invitations/send
email: "jmartinez@outlook.com"
referenceId: "10267" ← 订单直接绑定
name: "Jennifer Martinez"
2
Live Chat 内直接展示追踪链接
API 返回含 referenceId 的唯一链接，直接渲染在聊天面板中，无需发送邮件
Chat 面板显示：
👉 点击此处在 Trustpilot 留下您的评价 ↗
trustpilot.com/evaluate/...?ref=10267
✅ 无需发邮件 · 用户点击即确认意图 · 转化率可追踪
3
用户提交评价，referenceId 永久绑定
评价写入 TP 数据库时自动附带 referenceId
无需 Seel 做任何匹配操作
4
Webhook 实时推送到 Seel
评价发布后秒级通知，附带完整元数据
event: "review.created"
referenceId: "10267" ← 直接对应
rating: 5 · authorName: "TechShopper99"
✅ 精确归因，置信度 100%
referenceId 直接对应，无任何推断 rating ≥ 4 → $5
📊 方案对比总览 — 同一计费结果，两种完全不同的实现路径
维度 🆓 免费版 💎 Trustpilot Business
数据获取方式 每日爬取公开 HTML 页面 Webhook 实时推送（秒级）
归因核心机制 姓名模糊匹配 + 订单号关键词 referenceId 一一对应，无需匹配
归因置信度 55–100%，不稳定 永远 100%
漏归因风险 有 匿名用户名无法匹配 无 referenceId 强绑定
用户触达方式 无主动触达，被动等待用户自发评价 Live Chat 内直接展示追踪链接，即时引导
链接点击追踪 ❌ 不可追踪 ✅ 展示 / 点击 / 提交全链路
实时性 次日（24h 延迟） 即时（Webhook 秒推）
工程复杂度 高（Scraper + 三重匹配算法） 低（API 调用 + Webhook 接收）
误计费风险 存在 模糊匹配可能误判 零风险 精确绑定
前提条件 无，开箱即用 商户需购买 Trustpilot Business 套餐
