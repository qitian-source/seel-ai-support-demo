3 · 谁做什么

🧑‍⚖️ 法务 审阅 & 签署
  • Registry每条规则:查什么、引用哪条、解释依据。这是签署对象。
  • Scope 决定哪些查、哪些不查(如 checkout 整块删、Seel 自家产品规则移出)。
  • 语言 & 免责禁用词、counsel 结尾语、intro 免责声明措辞。
  • 复审模型重签周期、抽检比例、新品类 gate。
  • 不归他们管软件、UI、管线、对接。
🛠️ 研发 建真系统
  • Registry 做成版本化文件YAML/JSON 放仓库。法务签署 = 一个 approve 的 PR;release tag = 扫描拉取的 corpus 版本。
  • 引用渲染 = 查表finding 按 rule_id 从签署 registry 取引用文字。模型绝不产生引用。
  • 扫描引擎LLM 把页面证据 → rule_id + 判定(verified/review/not-assessable)+ 证据摘录。仅此。
  • Grounding / 防幻觉 CI无证据、或引用了不存在的 rule_id → 拒渲染;幻觉引用直接 build 失败。每次 corpus 改动跑 honeypot 回归。
  • 覆盖对齐UI 计数的每个 check 都必须是 registry 一条、且报告里可点开 —— 不能有"查了 58 项"却没登记(Michael #1)。
  • Evidence store每次扫描的页面快照 + 时间戳 + network trace —— 支撑 Proof 层与 provenance。
  • 对接 & 摄入USPTO(商标)、Middesk(SoS DE/CA/NY/TX)、corpus 从 EUR-Lex / eCFR / uscode / leginfo / FTC / OEHHA 摄入 + 月度链接健康检查。
🧭 产品 负责体验
  • Onboarding授权勾选 + DNS 域名验证 + 品类确认。
  • 报告体验同一 finding 的 Overview / Review-Items 双投影、Source/Proof pane、状态追踪、评分展示。
  • 品类检测 UXdetected / possible / not-detected 确认流。
  • demo(已完成)完整体验的可交互 mock —— 研发照它建。

4 · 现在在哪 → 什么解锁 launch

关键路径

✅ Demo —— 已完成
完整体验以 mock 形式上线,产品侧已定义。
◐ Registry —— 法务审阅中
Michael 审内容:覆盖、引用、措辞。法务的当前任务。
→ 研发 —— 建真系统
把 mock 变成上面那条管线(registry 文件、引擎、grounding 门、对接)。非法务的部分现在就能并行开工。
→ Launch
corpus 签署 + 研发建成后。corpus 没签,任何东西都不上商家。
唯一卡住全局的交界面:registry 就是法务与研发的边界。法务定规则和引用是什么;研发建一个只渲染签署过内容、绝不自己编法律文字的系统。registry 一签,build 和 launch 同时解锁。
