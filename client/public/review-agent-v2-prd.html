3
MD 页面改造（核心交付）

改造集中在 merchant-dashboard src/pages/review-agent/index.jsx，分五组：

3.1 页面结构：Performance / Settings 双 tab

Tab内容权限点
Performance时间范围 → Big number KPI 行 → 转化漏斗 → Invited Sessions 表（元素结构对齐 Sales agent Analytics：时间筛选复用 MD 全局时间控件——Yesterday / Last 7 / 30 / 90 days / Custom range From-To + Apply；指标大数卡横排含环比，下方图表与明细）review_agent:performance
Settings单张 Setup 卡承载全部配置：三步各自内联配置项（TP URL / Daily Limit / 总开关），步骤行展开即填写，不设独立配置卡片review_agent

配置与数据分离，权限点与 tab 一一对齐（解决 v1 "有 review_agent 无 performance 时整页 No permission" 的问题）。Support agent 的 widget 渠道设置中加 "Review Agent →" 入口链接做发现性；开关本体只存在于 Settings 一处，不做双开关。关闭 agent 不隐藏历史数据：Performance 顶部显示"已暂停发送"提示条，漏斗与列表照常可见。

3.2 数据口径（Performance）

Big number KPI 行（对齐 Sales agent 的 Impressions/Clicks/CTR/Orders/CVR/Revenue 六卡结构）：Invites Shown · Invite Clicks · CTR · Reviews · Review Rate · 5-Star Reviews，每卡含"±X% vs previous"环比（绿涨红跌）与定义 tooltip。

v1 指标v2 改为定义（tooltip 展示给商家）
Invites SentInvites Shown邀请卡在 widget 实际展示的会话数——无"发送"动作，Sent 会被误解为邮件
（无）Invite Clicks点击跳转 TP 的数量 + CTR（÷ Shown）
ReviewsAttributed Reviews归因成功的评价数 + review rate（÷ Clicks）
5-Star Reviews不变+ 占比（÷ Attributed Reviews）
📐 Cohort 口径（关键）：漏斗按邀请展示时间（invitedAt）分组，下游点击/归因事件计入其邀请所在 cohort，而非事件发生时间——否则跨窗口归因会导致转化率虚低或超 100%。归因窗口 12h（§5 口径参考），最近 12 小时内点击的邀请天然未成熟：漏斗下方常驻注记"最近 12h 点击仍在归因窗口内（N 条 pending）"。

小样本保护：分母 < 20 时不显示百分比，降级为原始数（如 "4/4"）并注明样本不足——杜绝 100% 5-star rate 假象。会话/邀请分离：表标题改为 Invited Sessions，计数与漏斗第一层严格相等，与 support-agent 的"会话"概念脱钩。

悬浮 hint：KPI 指标、漏斗各层、Outcome badge / 筛选 chips、置信度 chip 均带 hover tooltip（即时悬浮气泡，非原生 title 延迟提示），提示语按各表定义——尤其 invited（展示未点击，停留态不过期）与 expired（点击后 12h 未归因）的区别必须在 hint 中讲清。

3.3 Invited Sessions 明细

#改造项说明
1Outcome 列修复 状态 badge（invited 灰 / clicked 蓝 / reviewed 绿 / expired 橙），替换 v1 错误渲染的星级；outcome 字段 API 已返回，v1 前端丢弃了
2评价回显侧滑 reviewed 行内星级 + 评价单行截断；点击打开侧滑：状态时间线、会话摘要（含"Open in Support agent" 次级链接，替代 v1 直接跳走）、评价全文、置信度 badge（High/Medium，匹配信号悬浮展示）、计费状态、View on Trustpilot
3搜索 + 筛选 搜索框（customerName / orderId / conversationId 模糊匹配）+ outcome 筛选 chips + 星级筛选，走 API 参数，服务端分页不变
4低星评价跟进 1–4 星归因评价：表内评价旁显示 ⚑ Needs follow-up 标记；侧滑内 Follow-up 区块提供 Reply on Trustpilot 外链 + Mark as handled（处理后标记消失）；同时通过 Notifications 模块推送低星评价提醒（24h 内公开回复保护 TP 形象）

3.4 Settings

#改造项说明
1Setup 单卡 Settings 仅一张 Setup 卡：三步（连接 TP 页 → 设上限 → 开启）既是引导 checklist 也是配置本体——步骤行展开即内联填写（步骤 1 = URL + 校验状态，步骤 2 = 预设档位 + 保存反馈，步骤 3 = 行内总开关），不设独立配置卡片
2TP URL 可达性校验 保存时后端实际抓取一次确认页面存在，失败返回明确错误——消灭"配错域名归因永远为 0"的静默失败。状态点语义收紧：已配置且最近爬取成功才亮绿（Verified · Last crawled X ago），未配置 / 抓取失败分别为灰 / 红
3未配置禁止开启 TP URL 为空或校验失败时总开关置灰，避免链路空转
4Daily Limit dailyInviteLimit = 0 显示 "∞ Unlimited"（绿色态），预设增加 Unlimited 档，消除 "Capped at 0/day"；非预设值展示实际数值；保存成功给明确反馈（Saved 提示）
5邀请卡文案 Setup 卡内新增 "Invite card message" 步骤：系统提供默认文案，商家可自定义（textarea 编辑 + 自动保存 + Reset to default 恢复默认）；步骤状态显示 Default / Customized；未自定义即用默认，不阻塞开启

3.5 工程项

config 提交改字段级 PATCH 语义（只传变更字段），消除 v1 全量 POST 的并发互相覆盖；前后端契约以本文档 §4 为准，移除 v1 前端对 5 种 response 形状的防御性解析。

4
API 契约变更

4.1 GET /md/open/aisupport/review-agent/summary

新增查询参数 dateFrom / dateTo（缺省 = 近 30 天）。统计按 cohort 口径：时间范围筛选的是 invitedAt，四层指标均为该批邀请的下游事件计数。

{
  "enabled": true,
  "dailyInviteLimit": 50,
  "tpPageUrl": "https://www.trustpilot.com/review/example.com",
  "tpPageStatus": "verified",  // 新增：verified | unreachable | not_configured
  "lastCrawledAt": "2026-07-14T06:00:00Z",  // 新增：状态点副文案
  "invitesShown": 630,  // 改名：原 invitesSent
  "invitesClicked": 187,  // 新增：漏斗第二层
  "reviews": 23,
  "fiveStarReviews": 18,
  "pendingAttribution": 12  // 新增：仍在归因窗口内的邀请数（未成熟标注）
}

4.2 GET /md/open/aisupport/review-agent/sessions

新增查询参数：

参数类型说明
searchstring模糊匹配 customerName / orderId / conversationId
outcomeenuminvited / clicked / reviewed / expired，逗号分隔可多选
ratingnumber按星级筛选（仅对 reviewed 生效）
dateFrom / dateTodate按 invitedAt 过滤

list 元素新增字段：

{
  // —— v1 已有字段不变 ——
  "outcome": "reviewed",  // invited | clicked | reviewed | expired
  "clickedAt": "2026-07-07T01:24:00Z",
  "reviewedAt": "2026-07-10T09:12:00Z",
  "rating": 5,
  "reviewText": "Amazing support, solved my issue in minutes...",
  "reviewUrl": "https://www.trustpilot.com/reviews/xxxx",
  "confidenceTier": "medium",  // high | medium（§5 口径参考）
  "matchSignals": "Timing (79 min) + content"
}

4.3 PATCH /md/open/aisupport/review-agent/config

由全量 POST 改为字段级 PATCH：只传变更字段（enabled / dailyInviteLimit / tpPageUrl / inviteMessage 任意子集），未传字段不动，消除并发编辑互相覆盖。dailyInviteLimit = 0 含义统一为 Unlimited；inviteMessage = null 表示使用系统默认文案。

提交 tpPageUrl 时后端同步做可达性校验（实际抓取一次目标页）：失败返回 422 + { reason: "tp_page_unreachable" }，不落库；成功则更新 tpPageStatus = verified。tpPageStatus ≠ verified 时提交 enabled = true 返回 422 + { reason: "tp_page_not_verified" }（对应前端"未配置禁止开启"）。

4.4 低星跟进

sessions list 元素增加 followUpStatus：needed（reviewed 且 rating ≤ 4，未处理）/ handled / null（5 星或非 reviewed）。标记处理：

// Mark as handled
PATCH /md/open/aisupport/review-agent/sessions/{sessionId}
{ "followUpStatus": "handled" }

低星归因评价落库时同步向 Notifications 模块投递提醒事件（review_agent.low_star_review），带评价内容与会话链接。

5
口径参考（后端已上线，非本版交付）
ℹ️ 本章为后端已上线逻辑的口径快照，供前端展示对齐用，不在本版开发范围。触发资格、发送控制、归因匹配均已按此规则运行。

5.1 触发资格与发送控制

会话终态 = agent_solved 且 AI 意图判定 = satisfied（复用 support-agent 意图体系，无 CSAT 弹窗）→ 依次过 90 天去重（email 归一）→ 灰度（rolloutPct）→ 每日上限（超限直接丢弃，无次日队列；dailyInviteLimit=0 = Unlimited）→ 展示邀请卡（进入 invited）。被丢弃的会话不写入 Sessions 表。

5.2 Outcome 状态机

invited —用户点击→ clicked —归因命中→ reviewed ✓ clicked —超 12h→ expired
状态定义
invited邀请卡已展示、用户未点击。停留态，不超时——未点击的邀请永远停在此
clicked用户点击邀请、跳转 TP，12h 归因窗口开启
reviewed归因命中（终态）。评价正文 / 星级 / 置信度 / 匹配信号落库。仅 5 星计费 $5
expiredclicked 后 12h 未归因（终态）。与 invited 的区别 = 有没有点击

5.3 归因匹配规则（GravaStar 实测校准）

项规则
归因窗口点击后 12 小时——10 条归因评价中 9 条在 3.5h 内提交（最快 6 分钟，最长 10.8h）
匹配基础conversation_id（邀评记录）与工单直接关联，在此基础上叠加信号
匹配信号① 时间接近度 ② 客户名吻合 ③ 评价内容与会话摘要语义匹配
置信度High 客户名吻合或间隔 ≤ 20min 且内容对应；Medium 窗口内 + 内容对应；低于 Medium 不归因
排除项纯产品评价、邮件客服评价（非 live chat）不归因
