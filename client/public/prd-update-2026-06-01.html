1信息架构与导航重构

这是本次迭代的主干。应用从「单页多面板」重组为按功能划分的顶部 Tab 体系，每个功能各自拥有 Operations / Settings 子标签，配置不再散落各处。

改造前

设置集中在一个全局 Settings overlay；各功能混在一起，渠道与数字员工配置缺乏清晰归属。

改造后

6 个功能顶部 Tab，各自带 Operations / Settings 子标签；右上角独立的全局 Settings 中枢做深链与状态汇总。

关键变更
  • 3f661cb 重构为 6 个功能顶部 Tab（AI Manager / Playbook / Live Widget / Email Inbox / Zendesk / Performance），引入共享组件 SubTabBar，每页自带 Operations / Settings 子标签。
  • 3f661cb 新增右上角固定的全局 Settings 中枢：深链进各功能的设置模块（渠道 + AI Manager），并显示实时状态徽标；移除旧的全局 Settings overlay。
  • 3f661cb AI Manager 吸收 Agent Config 为其 Settings 子标签；渠道页在未连接时默认落到 Settings。
  • 25f9f0d 侧边栏把 Conversation 从主导航组移入 WORKFORCE 分组首位（位于 Support / Sales / VOC / Review agent 之上），主导航组现止于 Reviews。
2渠道与模式
  • 3f661cb 渠道模式统一为单一 ChannelMode：off / training / production，三个渠道（Live Widget / Email / Zendesk）共用同一语义。
  • 3f661cb Zendesk 工单收件箱复用参数化的 EmailInboxView，避免重复实现；渠道未连接时默认进入 Settings。
  • 25f9f0d 抽出独立组件 ConversationsView，支持可选入参 lockedChannel。
会话表格复用 · lockedChannel 入参
改造前

会话表格内嵌在 PerformancePage（约 265 行）；Live Widget 的 Conversations 子标签是空占位。

改造后

抽离为 ConversationsView；Live Widget 传 "Live Chat Widget" 锁定渠道，Performance 不传保持全渠道筛选——单一数据源。

3数字员工配置
  • 3f661cb 多 Rep 注册表 + 按渠道分配（per-channel overrides）：数字员工可在不同渠道继承或覆盖默认配置。
  • 3f661cb 新增 LiveChatWidgetCustomizer：聊天组件外观与升级（escalation）定制，升级策略走继承体系。
  • 3f661cb → 25f9f0d 配置归位：AI Manager 先吸收 Agent Config 为 Settings 子标签，本次会话进一步迁出为全局 Settings 的 Rep Config 子标签，AI Manager 回归「纯操作」。
Settings 子标签承载内容入口
ChannelsLive Widget / Email Inbox / Zendesk 三个渠道的设置跳板，展示各自连接状态（Live / Training / Connected / Not connected）。默认子标签
Rep ConfigAI 数字员工配置（ConfigureAgentSection）内联托管：人设、权限、共享默认值。深链跳转
深链机制：新增一次性意图标志 agentSettingsIntent。「编辑人设」或 Onboarding 的「Create an AI rep」带着该标志进入 Settings，自动定位到 Rep Config 子标签后立即复位。
4Playbook / 规则 本节重点

Playbook 从静态展示升级为可操作的规则中枢，覆盖规则的创建、审批、解析与溯源全流程。

规则全生命周期
  • 3f661cb 新建（NewRuleDialog）、编辑、删除全链路；状态文案 "Rule created / updated / deleted / accepted and live"。
提案审批流
  • 3f661cb ProposalReviewSheet：把来自工单的规则提案（Team Lead proposal）经审批后落地为 live 规则，支持「Edit rule before saving」。
文档解析模型
  • 3f661cb 新增 ParsedModule / ParsedTopic：文档解析为 modules → topics → rules 的层级；DocumentDetailSheet 展示解析结果。
来源溯源与组织
  • 3f661cb 规则来源区分：Team Lead proposal / Manager edit / Uploaded document。
  • 3f661cb 列表排序（Newest / Oldest）与按 module 过滤；示例 module：Refund Processing、Return Window、Seel Protection Claims、Shipping Delays。
价值：规则不再是只读文案，而是一条可追溯（谁提的、从哪来）、可审批、可编辑的运营资产，承接了「Team Lead 提案 → Manager 审批 → 上线生效」的真实闭环。
5Onboarding 引导优化
  • 3f661cb 引导步骤通过 goToChannelSettings / goToManagerSettings 深链跳转到对应设置模块。
调整 Connect a channel 的跳转目标
改造前

直接深链进 Live Widget 渠道设置——默认替商家选好了「Chat」渠道。

改造后

25f9f0d 跳转到 Settings → Channels 渠道选择列表，由商家自己挑选要接入哪个渠道。

调整 Import Policies 文案
  • 25f9f0d 步骤标题改为 「Import Policies (optional)」，把「可选」写进标题，任何状态下都可见。
  • 25f9f0d 移除原先仅在未完成时出现的独立「Optional」徽标，避免与标题语义重复；「Done」徽标在完成后照常显示。该步骤始终不计入 essentials 进度。
