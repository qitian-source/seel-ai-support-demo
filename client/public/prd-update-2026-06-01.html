Seel AI Support · 产品需求文档
定位：AI Manager = 数字员工管理中枢。让商家以最低门槛把 AI 数字员工部署到多渠道客服，并能可控上线、可运营规则、可溯源效果。本文从背景、目标、用户场景到功能需求与验收完整描述本期范围。

1背景与问题

Seel AI Support 让商家用 AI 数字员工跨渠道（Live Widget / Email / Zendesk）自动处理客服。早期版本配置分散、上线门槛高、上线后缺乏即时反馈，商家难以「快速跑起来并看到效果」。

待解决的核心痛点
  • 上线门槛高、路径不清晰：必须先创建 AI rep 才能上线，新用户卡在配置环节。
  • 信息架构混乱：设置集中在一个全局 overlay，渠道与数字员工配置缺乏清晰归属。
  • 上线后「看不到对话」：完成上线后没有直达实时会话，缺乏正反馈。
  • 限时政策无法表达：大促 / campaign 等有时间边界的政策，规则无法设定生效窗口。
  • 销售与对话割裂：转化订单无法回溯到驱动它的那一段对话。
2目标与非目标
目标 (Goals)
  • 降低上线门槛：连接任一渠道即可上线，不强制创建 rep。
  • 清晰信息架构：6 功能 Tab + 全局 Settings 中枢。
  • 上线即见效：go-live 后直达会话页。
  • 规则可运营：全生命周期 + 提案审批 + 文档解析 + 限时政策。
  • 风险可控：Live Widget production 灰度上线。
  • 效果可溯源：转化订单关联销售会话。
非目标 (Non-Goals · 本期不做)
  • 多 Rep 注册表 + 按渠道分配（per-channel overrides）—— 本期为单数字员工。
  • 多人设意图路由 + 中途转接。
3目标用户与场景
角色 (Personas)
  • 商家管理员 / AI Manager：配置渠道、设定模式与灰度、上线、查看效果。
  • Team Lead：审批来自工单的规则提案、监督会话质量。
用户故事 (User Stories)
  • 作为商家，我想连接一个渠道就让 AI 上线，以便快速试用、看到效果。
  • 作为商家，我想上传 SOP 文档自动变成规则，以便不必手写规则。
  • 作为商家，我想给大促政策设时间窗口，以便活动结束后规则自动失效。
  • 作为商家，我想在全量 production 前先灰度放量，以便控制风险。
  • 作为Team Lead，我想审批来自工单的规则提案，以便规则可控上线。
  • 作为商家，我想从转化订单点进对应对话，以便核对 AI 推荐的效果。
4功能需求与验收 本节重点

下列需求以「功能域」组织；每条含描述与验收标准 (Acceptance Criteria)，供研发与 QA 对齐。技术实现细节见 附录 A。

FR-1 · 信息架构与导航

6 功能顶部 Tab + 全局 Settings 中枢；渠道 Tab 按连接状态展示

应用重组为 AI Manager / Playbook / Live Widget / Email Inbox / Zendesk / Performance，各页自带 Operations / Settings 子标签；右上角固定的全局 Settings 中枢做深链与状态汇总。

验收标准
  • 默认仅显示 AI Manager / Playbook / Performance + Settings。
  • 连接某渠道后，对应渠道 Tab（Live Widget / Email Inbox / Zendesk）才出现。
  • 侧边栏 Conversation 位于主导航组、Reviews 之下；WORKFORCE 分组以 Support agent 起始。
FR-2 · Onboarding 与上线 (Go-live)

3 步上线，门槛仅依赖已连接渠道；每步完成引导 + 上线后直达会话页

Get Started 由 4 步精简为 3 步：连接渠道 → 导入政策（可选）→ 上线。每步完成给出英文 Toast 并自动跳转，已完成卡片提供跳转 CTA。

验收标准
  • 进度仅计 essentials（连接渠道、上线）；导入政策标注 (optional) 不计入。
  • 连接渠道即满足上线门槛，无需创建 AI rep 即可 go-live。
  • 每步完成弹英文 Toast 并自动跳转到下一步；已完成卡片有 CTA（Manage channels / View parsed rules）。
  • go-live 后自动跳到已连接渠道的 Conversations / Inbox 子标签——即使当前停留在该渠道的 Settings 也强制切换。
FR-3 · 渠道模式与灰度上线

统一 off / training / production；Live Widget production 支持灰度比例

三个渠道共用同一 ChannelMode 语义。Live Widget 切换到 Production 时弹确认框，可设定灰度（Rollout）比例。

验收标准
  • 三渠道模式语义统一：off（暂停）/ training（仅草拟）/ production（自主回复）。
  • Live Widget 选 Production → 弹 Switch to Production?，含滑块 + 预设（10/25/50/100%）。
  • 确认后 Mode 下方常驻显示当前灰度比例，可随时 Adjust。
  • Email / Zendesk 切换模式不弹灰度框。
FR-4 · 会话运营

Zendesk 会话页与 Live Widget 对齐；点击行查看完整会话

Zendesk 的 Tickets 子标签改为 Conversations，复用共享的会话表格组件（锁定 Zendesk 渠道），与 Live Widget 一致。

验收标准
  • Zendesk 子标签为 Conversations，列与 Live Widget 一致（Ticket / Customer / Intent / Outcome / Mode / Turns / Summary / Time）。
  • 点击任意行右侧滑出完整会话详情（Thread / Reasoning）。
FR-5 · Playbook 规则与文档（含限时政策时间规则）

规则全生命周期 + 提案审批 + 文档解析 + 限时政策时间窗口

Playbook 从静态展示升级为可操作的规则中枢；文档上传自动解析为 modules → topics → rules；若识别为限时政策，弹出时间窗口设置。

验收标准
  • 规则可新建 / 编辑 / 删除，状态文案正确（created / updated / deleted / accepted and live）。
  • 工单规则提案经审批后一键落地为 live 规则，支持「Edit rule before saving」。
  • 上传文档：Processing → Processed，解析出 modules / topics / rules，并区分来源（提案 / 编辑 / 上传）。
  • 限时政策：AI 识别为限时政策（大促 / campaign，如 BFCM、11.11）时弹出时间窗口设置（英文日期时间选择器 · Start / End）；保存后文档常驻显示生效窗口；规则仅在窗口内对 AI 生效，窗口外自动暂停。
FR-6 · Sales Agent 会话溯源

转化订单关联销售会话

每笔转化订单展示对应的会话 ID，点击可查看驱动该订单的完整销售会话。

验收标准
  • 每笔 Converted Order 显示会话 ID（SC-xxxx）与「View conversation」入口。
  • 点击侧滑显示销售会话线程：客户提问 → AI 推荐商品 → 下单（含订单 / 渠道 / Touchpoint / 金额等元信息）。
FR-7 · 数字员工配置（单 Rep）

单个 AI 数字员工：人设 + 权限，配置于全局 Settings → Rep Config

本期为单数字员工模型。数字员工的人设、语气与读写权限在全局 Settings 的 Rep Config 子标签内联托管。

验收标准
  • Rep Config 可编辑该 rep 的人设 / 语气 / 权限。
  • 不提供多 rep 管理与按渠道分配（见 Non-Goals）。
FR-8 · Performance 复合图

「Tickets & Resolution Rate」复合图：工单量柱状图 + 解决率折线

Performance Dashboard 将原单一面积图升级为复合图，同时呈现每日工单量与解决率趋势，便于关联「量」与「质」。

验收标准
  • 图表为复合图：柱状图 = 每日 tickets 数量（左轴），折线 = 解决率趋势（右轴，%）。
  • 含图例（Tickets / Resolution Rate），双 Y 轴刻度正确。
5边界与异常 (Edge Cases)
场景预期行为
未连接任何渠道三个渠道 Tab 全部隐藏；Go live 步骤锁定，提示先连接渠道。
文档解析失败文档进入 Error 状态并提供 Retry。
限时政策窗口非法开始时间 ≥ 结束时间时报错，阻止保存。
灰度比例为 0%当前允许（演示），建议后续加下限提示，避免「上线但无人可见」。
go-live 目标渠道未连接跳转兜底回 AI Manager，不落到隐藏 Tab。
规则提案被拒不落地为 live 规则，保留在提案列表。
待补充：限时政策的时区处理（按店铺时区 vs UTC）与灰度下限（避免 0% 上线）尚未定义。
6非功能需求 (NFR)
  • 国际化 (i18n)：面向用户的文案以英文为主；时间选择器采用自建英文日期时间控件，规避浏览器 locale 导致的本地化格式。
  • 隐私与安全：邮箱连接密码仅存于组件本地状态，不写入全局状态、不持久化；Demo 不收集真实用户数据。
  • 性能：纯前端 Demo，本期无新增运行时依赖。
  • 兼容性：现代 Chromium / Safari；预览基于 Vite dev server。
  • 可访问性：交互控件应键盘可达（当前为待完善项，见风险）。
