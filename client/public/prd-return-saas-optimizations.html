1 · 背景与商户反馈

两家 Return SaaS 商户在使用过程中提出了 5 个具体痛点，均集中在 Issues 列表的仓库核对效率 和 Rules Flow 编辑器的可读性 上：

bassdash（bassdash-fishing-gear.myshopify.com）· 渔具类目
① 希望 Issues 列表增加展示 tracking number 字段；② 希望支持导出 RMA 明细（RMA、order number、return item SKU、label tracking number、return reason），以便与仓库线下核对，而不是一个个点开，当前操作路径冗长；③ 消费者提交的 return item 希望展示 SKU，方便识别；④ 状态为 completed 的 RMA 仍希望能查看 return label & tracking number。
aelfriceden（aelfriceden0.myshopify.com）· 服饰类目
MD → Issues → Rules → Return rules 模块：较长的 multiple choices 选项目前默认只展示前几个字符，希望全部展开或做成鼠标 hover 效果，让商家可以看全界面。

1.1 问题本质

  • 仓库核对场景缺失：Return SaaS 当前的列表和导出都是 RMA 粒度（ReturnList 8 列 + exportReturns 汇总），而仓库收货是包裹 + 商品粒度——核对必须拿到 SKU 和运单号，目前只能逐个点进 /return/detail/:rmaNumber，每单 3~4 次点击。
  • 信息生命周期断档：RMA 完结（completed）后 label / tracking 入口消失，但仓库对账、物流理赔往往发生在完结之后。
  • Flow 画布可读性：Rules/Flow 基于 @xyflow/react，边标签（退货原因、multiple choices 选项）截断为几个字符，服饰类商户的选项天然很长（"The waist of the pants is too tight or too loose"），画布失去自解释能力。

3 · 详细设计

R1 · Issues 列表新增 Tracking # 列

  • 列位置：Status 与 Fraud risk 之间；展示 return label 的运单号（即 Seel 生成或商家上传 label 对应的 tracking number），带承运商角标（USPS / UPS / FedEx…）。
  • 交互：单元格内提供 一键复制（复制去空格的原始单号）；点击单号可跳转承运商轨迹页（复用详情页已有的 tracking link 逻辑）；复制/跳转均 stopPropagation，不触发行点击进详情。
  • 空态：label 未生成或未揽收扫描时显示 — Label not scanned 置灰文案。
  • 多包裹（一个 RMA 多个 label）：列内展示第一个 + +N 角标，hover 展开全部；导出时逐行展开（见 R2）。
  • 搜索框同步支持按 tracking number 精确查询（placeholder 更新为 RMA# / Order / Customer / Tracking#）。

R2 · Item 级明细 CSV 导出

现有 Download CSV 弹窗升级为两种导出格式单选，默认记住上次选择：

格式粒度字段
RMA 汇总（现状）1 RMA = 1 行维持现有字段不变，保证存量商户导出模板兼容
Item 明细 NEW1 退货商品 = 1 行rma_number · order_number · sku · item_name · variant · qty · return_reason（含 multiple choice 选项全文）· label_carrier · label_tracking_number · resolution_type · rma_status · created_at
  • 应用当前列表的全部筛选条件与日期范围，与列表所见一致。
  • 沿用现有异步导出机制：生成完成后发送下载链接至商户邮箱；预估行数 < 500 时同步直接下载。
  • 多商品 RMA 每个 line item 一行；同一 RMA 的公共字段（tracking 等）在每行重复，便于仓库用 Excel 直接 VLOOKUP。
  • return_reason 输出消费者实际选择的完整选项文案；有附加说明（Others 填空）时拼接在后，格式 选项 — 补充说明。

R3 · Return item 展示 SKU

  • RMA 详情页 Return items 卡片：商品名 / 变体行下方新增 SKU chip（等宽字体、可复制）。数据源为 Shopify line item 的 sku 字段；SKU 为空时回退展示 variant_id，再空则隐藏。
  • 覆盖同一组件的所有消费方：RMA 详情、Process items 弹窗、Packing Slip PDF（新增 SKU 列）。
  • 消费者端（return portal）不在本期范围，仅商户侧展示。

R4 · Completed RMA 保留 label & tracking 入口

  • 现状（源码定位）：ReturnDetail/components/ShippingInfo.jsx 中 Shipping documents 区块由 {isNeedReview && renderShippingDocuments()} 门禁——仅 needReview（In progress）状态渲染，RMA 进入 completed / expired / cancelled 等状态后 label 下载与 packing slip 入口消失。
  • 目标：只要 label 生成过，任何状态下该区块永久可见——包含 View label (PDF)、tracking number（可复制）、Track 跳转。
  • 终态下该区块降级为只读（不可重新生成 / 替换 label），并保留 label 来源标识（MERCHANT_UPLOAD / SEEL_GENERATED）。
  • Label 文件保留策略：与 RMA 数据同生命周期，不做提前清理。

R5 · Rules Flow 选项文案全文展示 + Hover

作用范围：Rules/Flow 画布内全部边标签——退货原因（Return reason → 下游）与 Multiple choices 各选项（→ Return / 下一节点）。

层方案
默认展示现状源码：Rules/Flow/Step/StepDefaultEdge.jsx 边标签为 max-w-[120px] + antd Typography.Text ellipsis 单行截断，定位在贝塞尔中点。改为全文展示、自动换行：max-width ~140px，最多 3 行（line-clamp:3）；白底 + 细边框，保证与连线交叠时可读。
Hover 浮层鼠标悬停任意边标签 → 深色 tooltip 展示完整文案（含超出 3 行部分），同时该条边高亮（brand 色加粗），辅助追踪路径。触屏设备点按标签等效。
布局适配标签变高/变宽后需两处适配：① dagre 布局（utils.jsx getLayoutedElements，现为固定 NODE 240×60、ranksep 120、nodesep 20）按标签实际尺寸重算间距；② 标签锚点从贝塞尔中点（t=0.5，扇出汇聚处横向间距过小必然压盖）移至靠近目标节点一侧（t≈0.8），相邻标签间距 ≈ 节点间距。切换展示不改动 flow 数据结构，纯渲染层改造，data2Flow/flow2Data 不变。
性能注意：大型 flow（服饰商户实测 60+ 节点）标签全量渲染后，初始 fitView 的可视密度会下降。默认缩放级别按内容 bounding box 重算；zoom < 0.5 时标签自动退回截断模式（LOD），避免小字糊成一团。

5 · 验收标准

  1. Issues 列表可见 Tracking # 列；有单号的 RMA 可复制、可跳转轨迹页；无单号显示占位文案；列表加载 P95 无可感知劣化（< +100ms）。
  2. 导出弹窗可选两种格式；Item 明细 CSV 中，多商品 RMA 正确展开为多行，SKU / reason / tracking 三字段与详情页一致；筛选条件对导出生效。
  3. RMA 详情、Process items、Packing Slip 三处均展示 SKU；SKU 为空时按回退规则处理、不出现空 chip。
  4. 任选一个 completed / expired RMA，Return shipping 区块仍展示 label 下载与 tracking，且为只读。
  5. Flow 画布中 ≥ 40 字符的选项默认完整可读（≤3 行）；hover 出现全文浮层且对应边高亮；60+ 节点 flow 无标签压盖、拖拽缩放流畅（60fps）。
  6. 回归：现状 RMA 汇总导出模板字段不变；flow 发布（flow2Data）产物与改造前逐字节一致。
