1 · 背景

1.1 商户反馈原文

bassdash（渔具类目，Shopify）
① 希望 Issues 列表增加展示 tracking number 字段；② 希望支持导出 RMA 明细（RMA、order number、return item SKU、label tracking number、return reason），以便与仓库线下核对，而不是一个个点开，当前操作路径冗长；③ 消费者提交的 return item 希望展示 SKU，方便识别；④ completed 状态的 RMA 仍希望能查看 return label & tracking number。
aelfriceden（服饰类目，Shopify）
Rules → Return rules 画布中，较长的 multiple choices 选项默认只展示前几个字符，希望全部展开或做成鼠标 hover 效果，让商家可以看全界面。

1.2 问题本质

  • 核对场景与产品粒度错位：仓库收退货是按「包裹 + 商品」核对的，而 Issues 列表和导出都是 RMA 粒度——商家要拿到 SKU 和运单号只能逐个点开详情，单量大时完全不可用。
  • 信息生命周期断档：RMA 完结后运单/面单入口消失，但对账、丢件申诉恰恰多发生在完结之后。
  • 规则画布不自解释：服饰类退货选项天然很长（尺码、部位描述），画布标签截断成几个字符后，商家无法确认自己配置的规则是否正确。

3 · 需求详述

R1 · Issues 列表展示逆向面单运单号（可下载）

作为商家运营/仓库，我在 Issues 列表就能拿到 Seel 打印的逆向物流面单运单号——复制去查逆向轨迹、对着入库退件，还能直接下载面单，不需要逐个点开详情。

现状问题

逆向运单号目前藏得很深，商家拿到它至少 3 步：点开 RMA 详情 → 找到 Shipping documents 区块 → Download label 下载面单 → 再从面单图片上人肉读出条码下方的运单号。查逆向轨迹和入库退件是仓库的高频日常动作，这条路径完全不可用。

需求描述

  • Issues 列表在 Status 与 Fraud risk 之间新增 Tracking # 列，展示 Seel 生成的逆向物流面单（return label）上的运单号（即面单条码对应的单号，如 USPS Ground Advantage 的 9300 开头单号），附承运商标识；商家自传面单的 RMA 同样展示其面单运单号。
  • 单元格两个行内操作，均不触发行点击进详情：
    • 一键复制运单号（复制去空格的纯单号，商家自行粘贴到承运商官网查逆向轨迹）；
    • 下载面单（PDF）——在最外层直接拿到面单，无需进详情。
  • 运单号不做承运商轨迹页跳转（v2.1 评审结论），纯文本展示。
  • 多包裹（一个 RMA 拆多个包裹寄回）：默认只展示第一个运单号 + +N 徽标，hover 徽标弹出浮层逐行列出全部包裹（Package 1 / 2…，各带承运商、单号、复制、面单下载），鼠标可移入浮层操作；行高不因多包裹变化。
  • 顶部搜索框支持按运单号查询（空格不敏感），placeholder 同步提示。

边界

  • 面单未生成或未揽收扫描：显示置灰占位 — Label not scanned。
  • 触屏设备：点按 +N 徽标等效 hover。

验收标准

  1. 列内运单号与该 RMA 面单条码上的单号一致（Seel 生成与商家自传两种来源均验证）。
  2. 有运单的 RMA 在列表内可复制、可直接下载面单 PDF（与详情页下载的为同一文件）；单号无外链；无运单显示占位文案。
  3. 双包裹 RMA hover +N 出现浮层，两个包裹的单号均可复制、面单均可下载；移出后浮层消失；行高与其他行一致。
  4. 搜索框输入运单号（带或不带空格）能命中对应 RMA。

R2 · 导出自选字段（含商品级明细）

作为商家仓库/运营，我导出一张 CSV：每行一个退货商品，带 RMA 号、订单号、SKU、运单号、退货原因，直接和仓库收到的包裹对着核。我还能自己挑要哪些列。

需求描述（交互对标 Shopify「Export orders」弹窗）

  • 列表右上 Download CSV 打开紧凑导出弹窗，弹窗内是一个字段复选列表：
    • 字段按 RMA / Item / Shipping label 三组分区、双列排布，显示人类可读名称（如 SKU、Tracking number、Return reason）；
    • 顶部 Select all（支持半选状态）+ 右侧已选计数（如 13 of 13 selected）；
    • RMA number 为必选，置灰不可取消，标注 Required；
  • 行粒度自动推导，商家无需选择格式：
    • 勾选了任一 Item 组字段（SKU / Item name / Variant / Quantity / Return reason）→ 导出为每个退货商品一行；
    • 未勾选任何 Item 组字段 → 导出为每个 RMA 一行（等价现有汇总导出；多包裹的运单号以 ; 拼接）；
    • 弹窗底部一行实时提示结果：Exports 9 rows — one row per return item / one row per RMA。
  • 不做 CSV 预览表（评审结论：保持弹窗简洁）。
  • Return reason 输出消费者所选选项的完整文案；有补充说明时拼接在后。
  • 交付方式：一律走邮箱发送（v2.1 评审结论）——点击 Export 即提交异步导出任务，弹窗关闭并提示已提交；文件生成后将下载链接发送至商户邮箱（弹窗底部明示收件邮箱）；不做浏览器即时下载。

字段清单

分组字段
RMARMA number（必选）· Order number · Customer · Created date · Resolution type · Status
ItemSKU · Item name · Variant · Quantity · Return reason
Shipping labelCarrier · Tracking number（商品行取其所属包裹的运单）

验收标准

  1. 全选导出：双商品 RMA 正确展开为两行，各行运单号与商品所属包裹一致。
  2. 取消全部 Item 字段后导出：行数等于 RMA 数，双包裹运单以 ; 拼接在一格。
  3. 列表筛选「Completed」再导出：文件只含 Completed 的 RMA。
  4. 取消 Select all：仅保留 RMA number；再全选恢复 13 列。
  5. 点击 Export：弹窗关闭并出现「已提交」提示，浏览器不发生直接下载；商户邮箱在生成完成后收到下载链接，文件内容与勾选一致。

R3 · 退货商品展示 SKU ID

作为商家，消费者提交的每个退货商品我都能直接看到该商品的 SKU ID，用自己的编码体系快速识别是哪个货。

需求描述

  • 此处展示的是商品的 SKU ID（即店铺后台商品变体的 SKU 编码，如 BD-WDR-CHM-42），而非商品名/标题。
  • RMA 详情页 Return item(s) 中，每个商品在名称/变体下方展示 SKU ID 标签（等宽字体、点击复制）。
  • SKU ID 同步出现在：Process item(s) 处理弹窗、Packing slip 装箱单、以及 R2 的导出字段中——凡是商家要「认货」的地方都有。

边界

  • 商品 SKU ID 为空时回退展示变体 ID；仍为空则不展示该标签（不出现空壳）。

验收标准

  1. 详情页 / 处理弹窗 / 装箱单三处 SKU ID 一致，且与商家店铺后台该变体的 SKU 编码一致。
  2. 点击 SKU ID 标签复制成功并有提示。
  3. 无 SKU ID 商品按回退规则展示，无空标签。

R4 · 完结 RMA 保留面单与运单入口

作为商家，RMA 完结后我做月度对账或向承运商申诉时，仍能回到详情页查看、下载当时的退货面单和运单号。

需求描述

  • 现状问题：RMA 进入 Completed / Expired / Cancelled 等终态后，详情页「Shipping documents」区块（面单下载、装箱单）不再展示。
  • 需求：只要面单生成过，任何状态下该区块永久可见——含面单 PDF 下载、装箱单下载、运单号（可复制）与轨迹跳转；多包裹按包裹分行展示各自面单。
  • 终态下该区块为只读：不可重发（Resend）、不可重新生成或替换面单。

验收标准

  1. 任选一个 Completed / Expired RMA：面单与运单入口可见、可下载、可复制。
  2. 终态详情页不出现 Resend 等修改类操作。
  3. Issues 列表 Tracking # 列对终态 RMA 同样展示（与 R1 一致）。

R5 · Rules 画布选项文案可读

作为服饰商家，我在 Return rules 画布上能直接读懂每条分支代表哪个退货原因/选项，不用逐个点开节点核对。

需求描述

  • 作用对象：画布中全部连线标签——退货原因、Multiple choices 各选项。
  • 默认展示：由现状「单行截断只剩几个字符」改为限长截断——约 40 字符处按词边界截断加省略号（不切半个单词、不留悬挂的括号逗号），标签最多两行、所有标签高度一致；短文案完整展示不截断。
  • Hover 看全文：悬停任意标签，深色浮层展示完整文案，同时该条连线高亮，便于确认标签属于哪条分支；移出即消失；触屏点按等效。
  • 评审结论：不做整段全文平铺（长文案全展开会导致标签互相压盖、画布可读性反而下降）。
  • 标签变为两行后，节点自动布局的间距随之适配，任意分支扇出处标签不得互相压盖。

验收标准

  1. ≥40 字符的选项默认呈现为两行内的词边界截断（带 …），无切半单词、无裁半行。
  2. hover 任意标签出现全文浮层且对应连线高亮；短标签同样支持 hover。
  3. 大型 flow（60+ 节点）下标签无压盖，缩放拖拽流畅。
  4. 该改动纯展示层：商家已发布的 flow 配置内容不发生任何变化。

R6 · 换货选品页加搜索框（模糊搜索）

作为消费者，我选换货时目录商品一屏屏翻很慢、也难找到想要的款/色/码——我直接在选品页搜商品名、颜色、类型甚至 SKU/条码，快速定位到要换的商品。

现状问题

换货商品选择（saas_return 的 exchangeSelection 步骤，即消费者端「Choose new item」页）只有一个商品网格，没有搜索。目录商品多时消费者只能滚动翻找，找不到就「Skip Exchange」放弃或将就选一个——换货转化和体验都受损。

需求描述（消费者端 · Choose new item 页）

  • 在页面标题「Choose new item」下方、商品网格上方，新增一个搜索框，placeholder：Search products。
  • 模糊搜索（对齐 Shopify 商品搜索能力）：不限于商品 ID/名称，匹配范围覆盖 Shopify 标准商品字段——title（商品名）· vendor（供应商）· product_type（商品类型）· tag（标签）· sku · barcode（条码）；这是 Shopify 通用的商品检索维度。
  • 匹配规则：输入即时过滤商品网格；按词命中、顺序无关——查询拆成多个词，每个词只要出现在上述任一字段即算命中（如 camo hoodie/hoodie camo 命中「Boxy CAMO Hoodie」；khaki 经 tag 命中卡其色裤；hoodies 经 product_type 命中带 Hood 但名称未含 "hoodie" 的款）；大小写 / 空格 / 连字符不敏感，支持部分词，无需完整。
  • 搜索框右侧有清除按钮（有输入时出现），一键清空恢复全部商品。
  • 无匹配结果时，网格区展示空状态文案 No products match "…". Try another keyword or SKU.
  • 搜索不改变选择逻辑：过滤后照常点选商品加入换货；已选商品在过滤/清空后选中态保持。

不做（本次范围收敛）

  • 不做「换不到就留言让商家配货」的方案（曾评估，v2.3 改为搜索方向）——因此无消费者留言区、无 exchangeNote 字段、无商家端换货留言 callout。
  • Continue 校验维持原样：必须选到换货商品才能继续（不引入「留言即可继续」）。

验收标准

  1. 选品页出现搜索框；输入关键词即时过滤出对应商品。
  2. 六字段全覆盖：分别用 title / vendor / product_type / tag / sku / barcode 的关键词都能命中对应商品（如 aelfric 命中该 vendor 全部、hoodies 命中该 product_type、khaki 命中该 tag、ae-hd-camo 命中该 SKU、0193458002059 命中该 barcode）。
  3. 按词命中、顺序无关：camo hoodie 与 hoodie camo 均命中同一商品；跨词/跨字段（如 purple hood）亦可。
  4. 无匹配时展示空状态文案；点清除按钮恢复全部商品。
  5. 过滤状态下选中商品，清空搜索后该商品仍为选中态。
  6. 未选任何商品时 Continue 置灰并提示 Select an item to exchange for.；选到商品后可继续。

4 · 非本期范围

  • 消费者端（Return portal）展示 SKU——本期仅商家侧。
  • 导出增加新数据字段（如仓库处置状态、退款金额明细）——按后续反馈排期。
  • Rules 画布节点编辑交互的其他优化（选项排序、批量编辑等）。
