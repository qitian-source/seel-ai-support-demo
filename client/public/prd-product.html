1产品概述
1.1 背景

电商商家面对的客服咨询高度同质化（物流查询 WISMO、退款、退货、地址修改、保障理赔等），却需要大量人力承接。传统工单系统只是「记录」工具，无法主动解决问题。Seel AI Support 的目标是把 AI 客服包装为一个可被「雇佣、培训、上岗、考核」的数字员工，让商家像管理团队一样管理 AI。

1.2 产品愿景

让每个商家都能在数分钟内「招聘」一名 AI 客服，把它接入现有渠道（在线聊天、邮箱、Zendesk），用自己的 SOP 文档「培训」它，安排它「上岗」（Training / Production），并通过 AI Manager 持续掌握表现、审批规则迭代——把客服从成本中心转为可规模化的自动化能力。

1.3 产品目标

快速上线

引导式 Onboarding，最快连一个渠道 + 建一个 Rep 即可上线，政策导入可选。

可信可控

Training/Production 双模式、读写权限、Guardrail、是否披露 AI 身份，全程可控。

持续进化

从真实工单中提炼规则提案，由人审批后沉淀进 Playbook，越用越准。

1.4 范围与非目标

范围内（In Scope）

多渠道接入与运营、数字员工配置与上岗、Playbook 规则/文档管理、表现看板与会话分析、引导式 Onboarding。

非目标（Out of Scope）

真实后端/接口、支付与计费实际扣款、跨租户管理、AI 模型训练本身；Sales / VOC Agent 为同平台兄弟产品，仅做概览。

2目标用户与角色
角色描述核心诉求
商家 / 客服负责人平台真实使用者，配置并管理 AI 数字员工。快速上线、可控、看得懂表现、能持续优化。
AI Manager（Alex）　系统角色产品内的「Team Lead」数字角色，带商家走 Onboarding、推送日报与规则提案、回答提问。把复杂配置转为对话式引导。
AI Rep（如 Ava / Max）　系统角色实际承接对话的数字客服，可多名，各有名字/人设/权限，按渠道分配。按 SOP 准确回答、合规执行动作、必要时升级。
人工坐席承接 AI 升级（handoff）的真人。清晰的升级上下文与交接说明。
3产品定位与核心理念

核心心智模型：AI Manager = 数字员工的管理中枢（Workforce Hub）。它贯穿数字员工的完整生命周期：

1
招聘 / 接入
连接渠道、创建 AI Rep
2
培训
导入 SOP，沉淀 Playbook 规则
3
上岗
设定 Training / Production 模式
4
考核 / 迭代
看日报、审批规则提案
设计原则：① 对话式引导优于表单堆砌；② 高频内容（会话、Playbook）入口浅，低频配置（Settings）收纳深；③ 每个上线动作都需明确的人类确认与可回退说明。

5核心概念与数据模型
5.1 渠道（Channel）与模式（Mode）

三类渠道：chat（Live Widget）/ email（Email Inbox）/ zendesk。每个渠道独立维护连接状态与运行模式 ChannelMode：

Off 关闭

已连接但 AI 不工作。

Training 培训

AI 只写内部草稿/备注，不直接回客户。

Production 生产

AI 直接回复客户。

派生状态：anyChannelConnected / anyChannelLive / isAgentLive（任一 Production）/ agentMode（取最高档）。setAllChannelModes() 可批量置档。Email 与 Zendesk 互斥为异步主干（asyncBackbone）。

5.2 数字员工（AI Rep）

多 Rep 注册表 reps: AiRep[]，每个含 id / name / color / personality / customTone / permissions / discloseAI。通过 channelReps: Record<ChannelKey, string[]> 把 Rep 分配到渠道（一个渠道可多 Rep，一个 Rep 可跨渠道）。primaryChannel 标记主渠道。

5.3 Playbook：规则 / 文档 / 主题
  • Rule：name / module / content / source / stats（used、avgCsat、deflection）/ versionHistory（配置历史）。
  • Document：上传的 SOP，解析为 modules → topics → rules 的层级结构，可启用/禁用。
  • Topic（提案）：type = proposal / question；含 badge（NEW RULE / RULE UPDATE）、currentRule/newRule、来源工单、status（pending/accepted/rejected）。
5.4 会话日志（ConversationLog）

记录完整对话与 AI 推理链（reasoning：上下文补全 → 规则路由 → 知识检索 → 动作执行 → Guardrail 校验 → 输出），并标注 intent、entry/exit sentiment、outcome、mode、channel、turns、duration、handoffNotes 等。是 Performance 与各渠道 Conversations 列表的统一数据源。

5.5 Onboarding 完成度（派生）
步骤含义完成判定必要性
Step 1连接渠道三渠道任一 connected必需
Step 2导入政策存在 Processed 且含 extractedRules 的文档可选
Step 3创建 AI ReprepHired必需
Step 4上线任一连接渠道处于 Training/Production必需

进度条仅统计 essentials（1/3/4），Step 4 在 1&3 完成前锁定（step4Status）。全部完成 → setupFullyComplete，AI Manager 从引导视图切换为日常运营视图。

6.1功能 · Onboarding 引导

未完成配置时，AI Manager 渲染居中的「Get Started」引导卡，含 4 步：

步骤文案/描述跳转目标
Connect a channel连接 Support Chat / Email / Zendesk，并指定各渠道由哪些 Rep 应答。Settings → Channels 渠道选择列表
Create an AI rep新建一个或多个 Rep，各有名字、人设与权限。Settings → Rep Config
Import Policies (optional)上传 SOP 提升回答准确率；随时可做，非上线必需。Playbook → Documents
Go live把各渠道 Mode 设为 Training 或 Production，让 Rep 上岗（1&3 完成前锁定）。主渠道 Settings + 上线引导气泡

Demo 控制：「Reset Documents」清空文档；「Complete All Steps」一键完成（连好三渠道、建 2 个 Rep、导入文档、完成分配）。进度条展示「N of 3 essentials」。

6.2功能 · AI Manager（运营中枢）
6.2.1 Team Lead 视图（Alex）

配置完成后呈现对话式信息流：

  • Channel Overview Panel：三渠道概览行（连接状态、地址/子域名、模式徽标、已分配 Rep 头像 / 「No rep」告警），点击进各渠道设置。
  • Daily Digest（日报）：3 个统计卡（处理工单数、自动解决率、升级率，含趋势）+「View full dashboard」链接 + 待审提案数提示。
  • Proposal Cards（规则提案）：每条来自真实工单的规则新增/更新建议，展示「What Changed」「New Rule」「来源工单」（可点开会话侧栏），Accept / Reject。
  • 自由问答：可向 Alex 提问，自动回复（Demo 模拟）。
6.2.2 Rep 视图
  • 头部含 Rep 头像/名称 + 「Go Live Mode」下拉（Production / Training / Off）+ Profile。
  • 升级队列（Escalation Feed）：需要人工关注的工单以对话卡呈现，工单号可跳 Zendesk，支持「Resolve」；已解决项置灰。空态显示「All caught up」。
  • 模式切换确认：有副作用时弹确认（例如 Production→Off 提示「3 个进行中工单将转人工」）。无连接渠道时禁用 Training/Production。
AI Manager 现为「纯操作」页面——所有配置（人设、权限、渠道连接）已迁移至全局 Settings，保持职责单一。
6.3功能 · 渠道运营
6.3.1 Live Widget（在线聊天）

子标签：Conversations（复用 ConversationsView，锁定 Live Chat Widget，展示不可移除的渠道上下文标签）/ Settings（连接表单、外观定制、聊天升级配置）。

6.3.2 Email Inbox

三栏布局：左（线程列表 + 状态筛选 New/Open/Pending/Solved + 搜索）｜中（线程详情 + AI 草稿，按模式着色：Off 灰 / Training 琥珀 / Production 蓝，可采纳/重生成）｜右（客户/意图/情绪/订单信息 + 升级）。支持基于发件人/关键词/情绪的 Flag Rules 自动标记。

6.3.3 Zendesk

子标签：Tickets（复用邮件式工单视图）/ Settings（5 步引导式连接：授权 → 选坐席 → Webhook → Trigger → 校验，每步带状态与错误反馈）。

6.4功能 · Playbook（规则与知识库）
Rules 子标签
  • 搜索 + 排序（Newest/Oldest）+「New Rule」；按 module 过滤标签（含计数）。
  • 规则卡列表（名称 + module 徽标 + 内容预览 + 来源 + 更新时间），点开 RuleDetailSheet。
  • 详情含：Content（可编辑）、Source、自动提取的 Actions、Stats（used / avgCsat / deflection）、Config History（版本时间线与 diff）。
  • 未完成 Onboarding 时顶部出现「N proposal(s) 待审」入口。
Documents 子标签
  • 空态：拖拽上传区（支持 PDF/PPTX/DOCX/TXT/CSV/XLS），自动提取规则。
  • 列表：文件类型徽标、大小/时间/状态（Processing/Error+Retry）、In use / Disabled 开关、删除。
  • AddDocumentDialog：上传文件 / 手动输入 两个标签页。
  • DocumentDetailSheet：按 modules → topics → rules 层级展示解析结果。
6.5功能 · Performance（表现看板）
Dashboard 子标签

顶部筛选：Time Range（Yesterday / 7d / 30d / 90d / Custom）+ Channel（All / chat / email / zendesk）。

  • KPI 卡：Total Tickets (156, +12)、Resolution Rate (68%, +4%)、Escalation Rate (32%, −4%)、Sentiment Improvement (54%, +6.2%)、Avg. Turns (4.5, −0.3)。
  • 图表：Resolution Rate 趋势（面积图）、Sentiment Flow（柱+双线，进/出情绪）、Sentiment Distribution（进/出双环形，Satisfied→Furious 五档）。
  • Contact Reason 表：Intent / Volume（条）/ Resolution Rate / Sentiment Change，按量排序、可下钻。
  • AI Summary：随筛选变化自动生成自然语言摘要（含生成态）。
Conversations 子标签

复用 ConversationsView（全渠道）：状态分段（All/Resolved/Escalated/Handling）、搜索、多维筛选（渠道/时间/结果/意图/情绪/模式）、排序，点开会话可见完整推理链侧栏。

6.6功能 · Settings（全局设置）
Channels 子标签

三张渠道跳板卡（Live Widget / Email Inbox / Zendesk），各显状态徽标（Live / Training / Connected / Not connected），点击深链至对应渠道页的 Settings 子标签。

Rep Config 子标签

内联托管 ConfigureAgentSection：选择/新增 Rep；编辑名字、人设（Friendly / Professional / Casual / Customize+自定义语气）、读写权限（含锁定项与 Guardrail）、是否披露 AI；并按渠道勾选分配该 Rep。

6.7功能 · Sales Agent / VOC Agent（兄弟产品概览）

Sales Agent

Analytics 标签：Time Range + 渠道模式多选；6 个 KPI（Impressions / Clicks / CTR / Orders / CVR / Revenue）；日趋势（柱+三线）；触点表现表，行可展开订单详情（含推荐商品高亮）。Touchpoints / Strategies 标签为 coming soon。

VOC Agent

「Ask Me Anything」输入 + 左侧工具栏（Overview/Trends/AI/Geo/Report）；品牌 Dashboard 卡（关键词、竞品、时间范围、导出）+ Market Insight Overview（全局摘要、增长引擎，术语可回填提问）。

7关键用户流程
7.1 首次接入到上线
  1. AI Manager「Get Started」→ Connect a channel → Settings/Channels 选渠道并连接。
  2. Create an AI rep → Settings/Rep Config 建 Rep、设人设/权限、分配渠道。
  3. （可选）Import Policies → Playbook/Documents 上传 SOP，自动提炼规则。
  4. Go live → 设渠道 Mode（建议先 Training 观察，再切 Production）→ setupFullyComplete。
7.2 日常运营闭环
  1. AI Manager 看 Daily Digest，掌握当日量与解决/升级率。
  2. 审批 Proposal（接受则规则沉淀进 Playbook，越用越准）。
  3. 处理 Rep 升级队列中需人工介入的工单。
  4. 到 Performance 下钻 intent / sentiment / 会话推理链，发现优化点。
8模式与状态机

模式切换需明确确认，并提示副作用（可回退/影响进行中工单）：

From → To提示
Production → Off有 3 个进行中工单，将被重新分配给人工坐席。
Production → Training进行中工单仍以 Production 完成；新工单走 Training。
Training → ProductionRep 将开始直接回复客户。
Off → ProductionRep 将立即开始直接回复客户。
无任何渠道连接时，Training / Production 不可选，引导用户先完成接入。
9权限与安全模型
  • 读权限（Read Actions）：查订单、查物流、查客户、查商品、查 Seel 保障、读工单历史、查退货状态等；多为默认开启、部分锁定。
  • 写权限（Write Actions）：回复、升级、取消订单、改地址、提 Seel 理赔、发起退货、退款、加内部备注等；含 Guardrail 约束，部分默认禁用，需显式开启。
  • 披露 AI 身份（Disclose AI）：可配置 Rep 是否向客户表明 AI 身份。
  • 升级与交接（Handoff）：可配置升级分组、坐席、标签、是否自动置优先级，附交接说明。
所有「写」类与上线类动作遵循「人类确认 + 可回退说明」原则，降低自动化风险。

11设计规范

模式色

Production = emerald；Training = amber；Off = gray。贯穿模式徽标与下拉。

情绪色（五档）

Satisfied #10b981 · Positive #14b8a6 · Neutral #94a3b8 · Negative #f97316 · Furious #ef4444。
