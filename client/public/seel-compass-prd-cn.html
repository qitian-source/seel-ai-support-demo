Seel Compass — 产品需求文档

给出海商家的合规基线评估工具。本 PRD 定义 Phase 1 要做什么、给谁、如何验收、如何度量成功。

1. 背景与目标

1.1 商家痛点

  • 合规工作找不到起点。商家想出海到 US / EU 时，不知道自己网站的哪部分需要改
  • 找当地律师贵。每个市场找一位合格律师，$50K–100K/市场是常态
  • 律师沟通效率低。商家不能给律师一个"哪里需要看"的 gap 列表，律师只能从零开始
  • 出海后再合规，代价成倍。已经发布后再改产品页、政策、退款条款，机会成本 + 律师纠纷成本极高

1.2 产品目标

  • 给商家一个客观、结构化的合规基线，标注哪些是 verified / review item / 无法核查
  • 让商家在找律师之前先做完自查，把复杂的合规工作变成 gap-driven
  • 每周自动 re-scan，跟踪合规状态变化
  • 把商家出海合规准备成本降低一个数量级

1.3 商业目标（Phase 1）

指标目标时间点
邀请制早期商家数5–10 个Phase 1 上线后 4 周内
Weekly active 商家≥ 80%（邀请制期间）Phase 1 上线后 6 周
付费模式预留 subscription 空间；Phase 1 免费邀请制Phase 2 讨论

2. 目标用户

2.1 主要用户画像

Persona：DTC 出海商家的运营负责人（Ops / Founder / eCom Manager）

规模：年 GMV $1M+，1–3 个海外市场准备/已进入

行业：outdoor / cosmetics / apparel / supplements / kids 等消费品垂直

技术能力：能熟练用 Shopify / Woo / BigCommerce 后台；不是律师、不懂合规术语

使用频率：Onboarding 时深度 review 一次；之后每周花 5–10 分钟看 delta

2.2 次要角色

角色关系使用方式
商家的外部律师接收商家转发的 review items只读 report；不直接登录 Compass
Seel 客户成功协助商家理解报告可能会代商家操作；帮商家解释 finding
Seel 法务（Michael / Jenny）负责 corpus + carve-out 签署 · 抽检 report 质量后台工具（不在 merchant dashboard 里）

2.3 典型使用场景

  • 场景 A —— 准备扩到 EU：商家已在 US，考虑进入 EU。想知道现有网站在 GDPR / EU 消费者法下有哪些差距
  • 场景 B —— 扩品前自查：商家新加了一条 cosmetics 或 supplements 线，想知道有没有踩到品类特有的合规要求
  • 场景 C —— 找律师前打好基础：商家马上要花钱找外部律师，希望能把范围缩小到 4–5 个具体 gap，而不是整站 review
  • 场景 D —— 持续监控：商家已完成一轮合规审查，想确保后续网站改动不引入新问题

3. 产品定位

一句话定位：你出海扩张与就绪度的伙伴。

类比：Compass 之于合规就像 SOC-2 readiness assessment 之于安全 —— 结构化的基线自查工具，不是法律意见，不替代律师。

核心区隔：市面上要么是律师（贵、慢），要么是模板化 SaaS 表格（浅、无法落地）。Compass 是面向出海商家的、基于扫描的、可追溯的合规基线工具。

3.1 明确非目标（must NOT）

  • ❌ 不是法律咨询、不给法律意见
  • ❌ 不是判决："某某违反了 XX 法" 这种话不会出现
  • ❌ 不替代律师 —— 每一条 review item 都以"建议与合格律师复核"结尾
  • ❌ 不做完整的 FTO (freedom-to-operate) 分析
  • ❌ 不审查商家的客服对话（Phase 2 候选）

3.2 产品语气原则

  • 用"待审项"（review item）而不是 "issue" / "violation"
  • 用"与……不一致"（inconsistent with）而不是 "违反"
  • 用"建议向合格律师确认"作为每个 review item 的结尾
  • 永不出现的词：violates · illegal · non-compliant · deceptive · unlawful

4. 核心用户故事

4.1 商家视角

US-1 · 商家 · 出海准备
作为一个准备扩到 EU 的商家，我想看到我的网站在 EU 隐私、条款、产品标示上有哪些 gap，这样我可以在花钱找 EU 律师之前先自己补上明显的问题。
US-2 · 商家 · 品类识别
作为一个卖化妆品 + 服装的商家，我想知道系统识别到了我卖化妆品，并且只对化妆品 SKU 应用 EU 化妆品法规检查，这样我不会为不相关的规则（比如酒类）浪费时间。
US-3 · 商家 · 追溯 evidence
作为一个不懂合规术语的商家运营，我想看到"为什么这一条被标为待审"背后的具体证据（比如"你的 /privacy 页第 4 节说了 X，但没提到 Y"），这样我能理解 gap 是什么，而不只是看到一个红标。
US-4 · 商家 · 转给律师
作为一个准备找律师的商家，我想把 Compass 生成的 review items 直接分享给我的外部律师，让律师从这个明确 gap 列表开始工作，而不是从零 review 我整站。
US-5 · 商家 · 持续监控
作为一个已经跑过一轮 Compass 的商家，我想每周自动收到"这周有什么变化"的报告 —— 新增了什么 gap、resolved 了什么、还有什么在跟踪，这样我不用手动去查。

4.2 Seel 法务视角

US-6 · 法务 · 一次性签方法论
作为 Seel 法务，我想一次性签署产品的方法论 + 法条 corpus，之后由架构保障机制（语言 filter、carve-out、审计日志）来管理每份 report 的输出，而不是每份都要我人肉过。
US-7 · 法务 · 抽检机制
作为 Seel 法务，我想能随机抽检 5% 的 report、并且能收到"高严重度 finding" 的自动通知，这样我能保持对产品输出质量的持续可见性但不用每份 report 都看。
US-8 · 法务 · Seel 自伤保护
作为 Seel 法务，我需要产品不会把 Seel 自己的售后保障 / opt-out 机制标为合规问题—— 因为那样等于 Seel 承认自家产品有问题、且给 plaintiff 律师留了书面证据。

5. 功能列表

Phase 1 的功能全景，按重要度排序：

🔒Data & Privacy 扫描

扫商家的隐私政策 + cookie 行为，产出 verified / review items（GDPR / CCPA / 各州 privacy laws）。

📄Terms & Conditions 扫描

扫服务条款、退款、担保措辞，识别 FTC §5 / Magnuson-Moss 相关的 gap。

🛍️Product 扫描（5 sub-groups）

Regulated categories / Claims / Safety / IP&Marketing / Consumer protection 五组，24 条检查项。

™️Intellectual Property 查重

USPTO 商标名查重，识别近似已注册商标，年度自动重扫。US only Phase 1。

🏛️Corporate Entity 状态

州 SoS good-standing 核查 + 年报截止日提醒。US only Phase 1，覆盖 DE / CA / NY / TX。

🤖品类自动识别 + 确认

Agent 扫商家 catalog 自动识别品类；商家在 Setup 里 confirm 或修改，用于决定 Product 模块的规则范围。

📋Review Items 集中视图

所有类目的待审项集中一个 tab，按 category 分组、可折叠；带状态追踪（Open / Ack / In remediation / Resolved）。

📈History 版本时间线

每次 scan 版本化记录，展示分数变化 · 新增 items · 已 resolve items。

🔄每周自动 re-scan

每 7 天自动重跑，商家收到变化通知；商家可主动触发 re-scan。

⬇️PDF 报告导出

整份 report 可导出为 PDF，方便商家转发给外部律师。

🚀Onboarding 五步引导（v1.1 新增）

连接店铺（含授权勾选）→ DNS 域名验证 → 选市场 → 发现扫描（实时进度）→ 确认品类；之后异步生成报告（约 10 分钟，邮件通知）。

⚙️Setup / 商家自服务

Store & markets · 品类确认 · Authorization + 域名验证状态。可自主启用新市场（UK / Canada 灰在 Phase 1）。

📊Merchant dashboard 集成

作为 Workforce 一个模块进入现有商家 dashboard，跟其他 agents 保持视觉一致。

6. 功能详细规格

6.1 Data & Privacy 扫描

要解决的问题：商家不知道自己的隐私政策 / cookie 处理在目标市场是否达到基线。

商家看到：一个 checklist（US 市场 62–64 条 / EU 市场 42–46 条），每条标注 verified / review item / not assessable。可点击展开查看 evidence（哪个 policy 页第几节命中了）。

典型 finding：Access/correct/delete/portability rights 是否描述 · lawful basis (GDPR Art. 6) 是否声明 · DSAR 回应时限是否说明 · non-essential cookies 是否 consent-before-set · retention statement 是否 enforceable（标为 Not Assessable）。

6.2 Terms & Conditions 扫描

要解决的问题：商家的 ToS / 退款 / 担保措辞在消费者保护法下有没有明显 gap 或过度承诺。

商家看到：~50 条 checklist（US）· ~39 条（EU），同样带 evidence pane。

典型 finding：governing law + 争议解决条款是否齐全 · warranty 措辞是否过绝对（Unqualified authenticity guarantee）· refund window 履行（标为 Not Assessable）。

6.3 Product 扫描（Michael 6/25 定义的 5 sub-groups · 24 条）

要解决的问题：商家产品页上的声明、标签、跨境合规是否有 gap，尤其是品类特有的强制要求。

Sub-group数量示例检查项
1. Regulated categories7Alcohol · CBD · Tobacco · Firearms · Supplements · Cosmetics · Pharmaceuticals
2. Claims & advertising4Health claims · Made-in claims · Green claims · Reference pricing
3. Safety, conformity & disclosures6Kids products · CE/UKCA · EU Responsible Person · INCI labeling · Prop 65 · Hazmat
4. IP & marketing integrity2Counterfeit / unauthorized reseller · Endorsement disclosures
5. Consumer protection & cross-border5Geo shipping restrictions · Sweepstakes · Comparative ads · COO disclosure · Urgency claims
Product 模块特殊逻辑：Sub-group 1（Regulated categories）依赖品类自动识别（§6.6）—— 商家不卖的品类完全不呈现规则评估，标为 N/A（"Not detected in catalog"）。这避免了信息噪音，也让"我们只审你实际卖的东西"这一原则可见。
关键 UX 约束（Michael 6/25）：Product section header 不能使用 "checkout-flow analysis is out of scope" 这种负向声明—— 会给商家和法律 opponent 一个 "Seel 明知这里有风险但选择不看" 的把柄。要用正向语言描述"我们做什么"。

6.4 Intellectual Property 商标查重

要解决的问题：商家在美国用自己的品牌名做营销之前，需要知道有没有已注册的相似商标可能构成风险。

商家看到：输入品牌名 → 系统在 USPTO 数据库里查找相似商标 → 显示：查询数据库 · 品牌名 · 找到的近似商标 · 持有人 · 建议下一步。

范围（Phase 1）：Name only（仅文字商标）· 不含 logo / 外形 · 不含 utility patent · 不含 FTO 分析

持续监测：商家可开启年度自动重扫。

与 Product #18 的区别：本模块查的是商家自己品牌名是否跟已注册商标撞车；Product #18 查的是商家 listing 里是否卖别家品牌的假货 / 灰市货。这是两个不同的 scan，商家 UI 里有交叉引用。

6.5 Corporate Entity 状态核查

要解决的问题：商家的美国实体是否处于 good standing，年报 / franchise tax 是否即将到期。

商家看到：Entity questionnaire（一次性填 formation state / entity type / registered agent）→ 系统查询州 SoS → 显示：Status · Next annual report due · Last checked。

Phase 1 覆盖州：Delaware · California · New York · Texas（~80% DTC 商家注册在这四个州）。其他州标为 Not Assessable + 引导手动上传申报文件。

6.6 品类自动识别 + 商家确认（Hybrid）

要解决的问题：Product 模块的规则需要按品类应用；但商家不应手填 40 个品类，也不应完全依赖 AI（有漏报风险）。

混合方案：

  1. Onboarding 发现扫描（§6.11 步骤 4）时 agent 扫商家 catalog，用 AI 识别可能的品类
  2. 识别结果按置信度分档展示给商家：≥ 90% Detected · 70–89% Possible · < 70% Not detected
  3. 商家在 wizard 第 5 步的确认页勾选或取消勾选（之后可随时在 Setup 里改），最终 set 用于报告生成和所有后续扫描
  4. 每周 re-scan 时增量重跑识别，如发现新品类会主动通知商家确认

冲突处理：商家 declaration 覆盖 AI 结果。冲突事件内部审计记录（法务可查），不呈现给商家。

6.7 Review Items 集中视图

要解决的问题：商家想要一个"所有需要我关注的事"的集中列表，而不是在 5 个类目里翻找。

商家看到：

  • 按 category（5 类目）自动分组
  • 每组默认展开、可折叠
  • 每条 review item 显示：标题 · 相关规则背景 · 检索来源 · 建议 · 状态追踪 pill（Open / Acknowledged / In remediation / Resolved）
  • 市场切换（US ↔ EU）会过滤显示对应市场的项

6.8 Review Item 状态追踪

状态转换：

  • Open（默认）—— scan 首次产生此项
  • Acknowledged—— 商家已看到并可能已分享给律师
  • In remediation—— 商家正在整改
  • Resolved—— 只能由下次 scan 确认规则不再触发时自动设置；商家不能手动 mark resolved
为什么 Resolved 不能手动标：防止商家把没真正修好的 item 标成"已解决"而误导自己/法务/未来审计。resolved 只能由 evidence 消失来证明。

6.9 History 版本时间线

商家看到：时间线视图，每次 scan 一个节点，显示：日期 · 分数 · +N new · N resolved · 具体新增/resolved 项目名。可对比任意两次 scan 的 diff。

6.10 PDF 报告导出

用途：商家常需要把整份 report 转发给外部律师、董事会、投资人。

要求：能一键从任意 tab 生成完整 PDF，print-friendly 排版，包含所有 tabs 内容 + footer disclaimer。

6.11 Onboarding 引导流程（v1.1 新增）

要解决的问题：首次进入的商家需要一条从"什么都没有"到"拿到第一份报告"的完整路径，且授权 / 验证两道门槛必须在任何扫描发生之前完成。

五个步骤 + 两个门槛：

  1. Connect store —— 输入店铺域名；必须勾选授权声明 "I confirm I am authorized to have this domain reviewed"，不勾无法继续（门槛 1：合同层）
  2. Verify ownership —— DNS TXT record 验证（seel-compass-verify=…，可复制），行业标准做法；未验证的域名不会被扫描（门槛 2：尽职层，防"替竞对扫描"的 paper-trail 风险）
  3. Target markets —— US / EU 可选（至少 1 个），UK / Canada 显示 coming soon
  4. Discovery scan（发现扫描） —— 快速轻量（约 2 分钟，分阶段实时进度）：爬取页面 → 读政策文档 → 编目商品 → 识别品类。爬虫排除集从第一次抓取就生效（checkout 页永不抓取）
  5. Confirm categories —— 识别到的品类预勾选并附 catalog 证据（如 "Sunscreen SKUs detected"），商家可增删；确认后的 set 即规则范围
两段式扫描（为什么报告不是立刻出）：确认品类后进入异步的报告生成阶段（应用市场规则集 + 编写 findings，约 10 分钟）。产品内显示 "Generating your first report…"，商家可离开页面，完成后邮件通知。这样重量级规则计算只跑一次、只对商家确认过的品类跑 —— 不会存在一份 scope 仅由 AI 识别（未经商家确认）决定的报告。

7. 用户流程

7.1 首次 onboarding（v1.1 · 与 demo 一致）

1 商家首次进入 Compass —— 五步 wizard + 异步报告生成
与 v1 的差别：v1 里"首次全量扫描 ~10–15 分钟"是 wizard 的最后一步、商家盯着进度条等；v1.1 拆成发现扫描（快，wizard 内实时进度）+ 报告生成（慢，异步，可离开等邮件）。品类确认从 Setup 页提前到 wizard 第 5 步，成为报告生成的前置条件。

7.2 日常查看流程

  1. 商家收到"新报告出来了"或"weekly delta"通知（邮件 / dashboard 内提醒）
  2. 打开 merchant dashboard → 左侧 Sidebar 点 Compass
  3. Overview tab 上一眼看：Overall 89% · 5 类目分数 · Since last review +2 / +2 new（D&P / T&C / Product 三个 section 默认折叠，标题行直接显示 verified / review 统计）
  4. 关心某类 → 点该 category card：section 自动展开并平滑滚动过去（IP / Entity 卡则切换到对应 tab）
  5. 或者点 Review Items tab → 看所有待审项集中列表
  6. 点开某条 review item → 看 evidence · rule context · 建议 · 状态 pill
  7. 标为 Acknowledged 或 In remediation
  8. 把这一条转发给自己的律师

8.3 关键 UI 元素约定

  • Score hero：6 张同尺寸卡片一行并排（Overall 89% + Data & Privacy 92 + T&C 86 + Product 88 + IP Review + Entity Good standing）。IP 和 Entity 卡右上带 "US" 徽标。
  • Section 默认折叠：Overview 的 D&P / T&C / Product 三个 section 默认收起，标题行显示 emoji · 类目名 · verified/review 统计 · chevron；点击标题展开/收起
  • 评分方法说明：summary 行内 "ⓘ How are these scores calculated?" 可展开评分口径（类目分 = verified ÷ assessed，Not assessable / N/A 不计入；Overall = 三个记分类目均值）—— 回应 Michael 6/11 "score 算法可查"
  • Category cards 可点击：点击卡片自动展开对应 section 并平滑滚动（Overview 内锚）；IP / Entity 卡切换到对应 tab
  • Onboarding wizard：顶部 5 步 step indicator（已完成 ✓ / 当前高亮 / 未到灰色）；授权 checkbox 未勾选时 Continue 禁用；DNS TXT record 可一键复制；生成中页有 spinner + "约 10 分钟 · 可离开 · 邮件通知" 提示
  • Checklist 项：每一条可点击展开 evidence pane，含"相关规则背景 / 已核查内容 / 检索来源 / 建议"四个区域
  • Review Items 分组头：展示 emoji · 类目名 · 数量 · chevron 折叠图标
  • Not Assessable 状态：灰色，配"omitted, not guessed"说明
  • N/A 状态：灰色，配"Not detected in catalog. Rules skipped."说明

8.4 法务视觉约束

硬约束：下列任何一项在 UI 里出现都视为 bug：
  • 使用 "violates" / "illegal" / "non-compliant" / "deceptive" / "unlawful"
  • 使用 "you must" / "must comply" 等指令式动词
  • 暴露"Seel-provided component" 标签
  • 暴露"checkout-flow analysis is out of scope" 之类的负向声明
  • Discuss / Chat tab 可见（即使是 disabled 状态也不行）
  • 缺少 footer disclaimer

8.5 品牌与视角

  • 视角：商家自己的工具（第二人称："your review" · "your catalog"），不是"我们在审你"
  • 不显示："Client portal" · 商家自己网站 URL（商家自己就知道）· "markets: US, EU" 冗余显示（下面市场 pill 已经有）

9. 验收标准

9.1 功能验收（每个 feature 都要满足）

AC-1 · Data & Privacy 扫描
  • 能对一个测试商家的 /privacy 页产生 verified / review / not assessable 三种状态的 findings
  • US 市场至少 60 项 · EU 市场至少 40 项
  • 每一条可点击查看 evidence（引用 policy 章节）
AC-2 · Terms & Conditions 扫描
  • 能识别测试商家 ToS 里的 governing law / dispute resolution / warranty 措辞
  • US 市场至少 45 项 · EU 市场至少 35 项
AC-3 · Product 扫描
  • 能对识别到的品类应用对应规则
  • 未识别到的品类清晰标为 N/A（"Not detected in catalog"）
  • 5 sub-groups 展示结构与 Michael 6/25 清单一致
AC-4 · IP 商标查重
  • 输入品牌名能返回 USPTO 数据库里的近似商标列表
  • 结果按 Nice 分类相关性过滤
  • UI 明示 "name-only, not FTO"
AC-5 · Corporate Entity 状态
  • 覆盖 DE / CA / NY / TX 至少四个州
  • 能查询 good-standing 状态 · 下次年报截止日
  • 非覆盖州显示 Not Assessable + 引导手动上传
AC-6 · 品类识别与确认
  • 系统对测试商家 catalog 能自动识别至少 3 个品类
  • 识别结果按 3 个置信度分档呈现（Detected / Possible / Not detected）
  • 商家能勾选 / 取消勾选并触发规则重跑
  • 识别置信度 ≥ 85%（在 7 类品类的标注测试集上）
AC-7 · Review Items 集中视图
  • 按 category 自动分组
  • 组内 items 数量对
  • 点击 header 可折叠 / 展开
  • 市场切换正确过滤 items
AC-8 · Review Item 状态追踪
  • 商家能在 Open / Acknowledged / In remediation 之间自由切换
  • Resolved 只能由 scan 触发（商家 UI 上不能手动点）
  • 状态变化在 History 中可追溯
AC-9 · Weekly re-scan
  • 每 7 天自动触发一次全量 re-scan
  • 产生 delta 通知给商家
  • History timeline 更新新条目
AC-10 · PDF 报告导出
  • 能一键生成完整 PDF
  • PDF 包含所有 tabs 的核心内容 + footer disclaimer
AC-11 · Onboarding 引导流程（v1.1 新增）
  • 授权 checkbox 未勾选时无法进入下一步；勾选事件带时间戳留存
  • 域名未通过 DNS TXT 验证时不触发任何扫描
  • 发现扫描分阶段实时进度展示（爬取 / 政策 / 编目 / 品类识别）
  • 品类确认页预勾选识别结果并附 catalog 证据；商家可增删
  • 确认品类后进入异步生成态：页面明示约 10 分钟 + 可离开 + 邮件通知；完成后报告在 Compass 下等待
  • onboarding 里确认的品类 set 与 dashboard 的规则范围一致（分数 / badge / N-A 项全联动）

9.2 法务硬性验收

AC-Legal（必须 100% 通过）
  • UI 里搜不到 "violates" / "illegal" / "non-compliant" / "deceptive" 等禁词
  • 每个 tab 都有 footer disclaimer
  • IP tab 有 "name-only, not FTO" in-context disclaimer
  • Entity tab 有 "informational only, confirm with counsel" 提示
  • 商家 report 里搜不到 "Seel-provided component" 或类似标签
  • Honeypot 回归测试：Seel 自营 demo 店 + 5 个匿名化 Seel 使用商家页 = 0 个 red finding
  • Michael + Jenny 已签署 corpus + skip-rule whitelist

9.3 性能验收

AC-Perf
  • 发现扫描（onboarding 内，含品类识别）P95 < 3 分钟
  • 首次报告生成（异步）P95 < 15 分钟 · 目标均值 ~10 分钟（与产品文案一致）
  • Weekly re-scan P95 < 8 分钟
  • Report 页首屏加载 < 1.5 秒

12. 依赖与风险

12.1 前置阻塞项（不解决无法上线）

  2. Honeypot 回归测试全绿（Seel 自营页 + 已知 Seel 商家页 = 0 red finding）
  3. Michael 抽检首 5 份对商家的 report 无 language / framing 违规

12.2 关键假设

  • Michael 6/25 提交的 24 items 相对稳定（Phase 1 内不会大幅重构）
  • USPTO ODP API 稳定可用
  • Middesk 服务对 DE / CA / NY / TX 覆盖度足够
  • LLM category classifier 在 7 类品类上准确率 ≥ 85%
  • 商家愿意在 onboarding 花 30 秒确认品类
